## 2. D4 — `provider-structured-json` guest walker

**Pure, in-guest (D4.1).** New module `providerStructuredWalker.ts` joins the guest
module set and bundles into `worker-bundle.cjs`; it runs under bare Node. The
orchestrator ships provider JSON OPAQUE (Outlook provider → `providerStructuredJson`
field, threaded unparsed through gateway/sync to the seam). Nothing in the
orchestrator inspects the structure.

**Convergence (D4.3).** The walker produces the SAME internal representation
(`ParseOut`) the RFC822 parser produces, extracted into a new shared module
`depackageModel.ts` (`Leaf`, `ParseOut`, the failure taxonomy, the C4 constants).
The join point is **before** SafeText derivation and carrier detection:
`emailDepackage.buildResultFromParse(parsed, pub)` is the single shared pipeline —
carrier detection (R3), the INV-7 ambiguity check, sealing, HTML→SafeText (R1), and
the typed result union are **not duplicated** per input form.

```
RFC822 bytes ─ hardenedParse ──┐
                               ├─► ParseOut ─► buildResultFromParse ─► DepackageEmailResult
Graph JSON ─ walkProviderStructured ┘   (R3 detect · seal · R1 derive · typed union)
```

**Untrusted structure + C4 guards (D4.2).** `walkProviderStructured` treats the JSON
as untrusted: bounds the raw payload against `maxInputBytes`, bounds JSON nesting
depth, parses defensively, and the Outlook adapter charges per-part size, part
count, and decoded-total against the same C4 limits as the RFC822 path; base64
`contentBytes` is decoded with the decode-ratio bomb guard. The adapter interface
(`ProviderStructuredAdapter`, an addressable registry keyed by provider) makes a
second provider schema an addition, not a rewrite.

**Failure behavior (D4.5).** Unrecognized/contradictory structure (non-object JSON,
non-string body content, unrecognized `body.contentType`, non-array attachments,
non-string `contentBytes`, unknown provider, parse failure) → `E_AMBIGUOUS_STRUCTURE`
→ quarantine (mapped to `email_depackage_ambiguous_structure`). Limit breaches →
`E_LIMITS_EXCEEDED` / `E_DECOMPRESSION_BOMB`. Never a partial best-effort result.

**Equivalence corpus (D4.4).** `providerStructuredWalker.equivalence.test.ts`:
the same logical message in both forms yields equivalent results. Corpus contents:

1. plain `text/plain` ↔ Graph `body{contentType:text}` → both `plain`, equal text, 0 artifacts.
2. HTML-only (R1 derivation + sealed HTML artifact) ↔ Graph `body{contentType:html}` → both `plain`, equal derived text, 1 sealed `text/html` artifact.
3. pure carrier (qBEAP as whole body) ↔ Graph `body{contentType:text}` → both `beap-carrier`, byte-identical package.
4. mixed: text body + `.beap` (pBEAP) attachment ↔ Graph `body` + `fileAttachment{contentBytes}` → both `mixed`, **byte-identical** extracted package, 0 sealed.
5. ambiguous: `.beap` attachment that is not a valid package ↔ same as Graph attachment → both `E_AMBIGUOUS_CLASSIFICATION`.

Body text is compared modulo trailing whitespace (MIME segmentation leaves a
trailing CRLF on the RFC822 plain part that the Graph string lacks; renderer/SafeText
treat trailing whitespace as insignificant). Packages, types, and artifact counts
are compared exactly. Plus a standalone ambiguous-structure failure suite (8 cases)
+ oversized + nesting-bomb. The corpus doubles as the V5 **dual-fetch** instrument
(see `0009` V5).

**Accepted narrow exception (reported, not absorbed silently).** Spec `0010` D5.1
says "every parse location removed or behind the flag-off branch." The IMAP/Gmail
provider parses are **retained flag-on for envelope metadata** (from/to/subject/date/
message-id), consistent with the B2 design accepted in `0008`; only their **body
output** is dropped flag-on. The fully-clean end state (envelope from the IMAP
`ENVELOPE` attribute / Gmail header metadata, with NO body parse at all) is a
follow-up — it cannot be validated off-rig (no live IMAP/Gmail servers) and is
flag-gated. The guard backstops it: any flag-on inline body-derivation or raw-byte
classification fails closed regardless.

### 3.2 The guard (D5.2)

`email/inlineParseGuard.ts`: `assertNoInlineParse(entryPoint)` throws the typed
`InlineParseForbiddenError` (`code = 'E_INLINE_PARSE_FORBIDDEN'`) iff the cutover
flag is ON; inert (one flag read) when OFF. Mapped to `email_depackage_inline_parse_forbidden`
in the quarantine table. **Always-on with the flag** (not test-only): in correct
flag-on operation the guarded points are never reached (the byte-courier path
replaced them), so the guard only ever fires on a regression / missed cutover
location — and when it does, it fails closed loudly instead of parsing untrusted
bytes. This is what makes V3's "the orchestrator provably never parsed the bytes" a
proof rather than an absence of logs.

Tests (`inlineParseGuard.test.ts`, all green, DB-free): flag-off → inert; flag-on →
typed throw with entry point; flag-on forced inline call (`viaSeam=false`) → fails
closed; flag-on seam re-entry (`viaSeam=true`) → guard does not fire.
