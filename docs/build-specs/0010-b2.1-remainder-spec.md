## D4 — The `provider-structured-json` guest walker (the missing safety net)

Build the guest-side walker that R2 designated as the universal fallback. Requirements:

1. **Pure, inside the guest payload** — it joins `depackagingWorker.ts`'s module set, bundles into the worker bundle, runs under bare Node. The orchestrator never touches the structure (it ships the provider JSON opaque, exactly as wired).
2. **Treats provider JSON as untrusted structure.** Schema-tolerant walking (Outlook Graph message schema first; design the walker interface so a second provider schema is an addition, not a rewrite). All C4 guards apply to this input form: nesting depth, per-part size, decompression/base64-expansion ratio, `limits.maxInputBytes` honored on the decoded total.
3. **Converges on the same internal representation** the RFC822 path produces before SafeText derivation and carrier detection — so HTML derivation (R1), the verbatim detection rule list (R3), the typed result union, and the failure taxonomy are shared, not duplicated. If full convergence is impractical, converge at the highest layer possible and document exactly where the paths join.
4. **Equivalence tests:** a fixture corpus where the same logical message exists in both forms (RFC822 and Graph JSON) must yield equivalent results — same derived text, same extracted packages byte-identical, same artifacts, same failure outcomes. This corpus becomes a V4/V5 instrument: live-account messages fetched both ways must match, which is a stronger Outlook fidelity check than comparing `/$value` to the provider's parse alone. Add that note to `0009` V5.
5. **Failure behavior:** unrecognized/contradictory structure → `ambiguous-structure` (or the closest existing taxonomy code) → quarantine. Never a partial best-effort result (INV-7).

## D5 — Residual orchestrator parse removal + guard instrumentation

1. Finish the Phase 2.2 byte-courier delta: every orchestrator-side parse location from the analysis's finding A is removed or sits strictly behind the flag-off branch. Produce the checklist (file:line → resolution) in the report.
2. Build the **guard** V3 depends on: when the depackage flag is ON, invocation of any inline-parse entry point (the MIME parser entry, the `gateway.ts` HTML derivation, raw-byte carrier detection in `messageRouter`) throws a typed fatal error (`E_INLINE_PARSE_FORBIDDEN`) that routes the message to quarantine via the mapping table. The guard is always-on with the flag (not test-only) — it is the runtime embodiment of invariant-0, cheap (a flag check at a handful of entry points), and it is what makes V3's "orchestrator provably never parsed the bytes" a proof rather than an absence of logs. Unit tests: flag-on + forced inline call → guard fires → quarantine; flag-off → guard inert, zero behavior change.
