## §1 — Guest display envelope

- **`displayEnvelope.ts`** (new, pure: `Buffer` + global `TextDecoder` only):
  - `decodeHeaderText` — RFC 2047 encoded-word decode (B/Q; per-word charset via
    `TextDecoder`; adjacent-word whitespace collapse per §6.2). Any encoded word that
    cannot be decoded (unknown charset, malformed) → the field degrades to the raw
    string, `degraded:true` — never a partial mix.
  - Address-list parsing (`splitAddressList` respects quotes + angle brackets, so a
    quoted-comma display name does not split), name decode + caps.
  - Date normalization (ISO 8601 when parseable; otherwise capped raw + degraded).
  - C4 caps in `ENVELOPE_CAPS` (subject/name/email/date length, recipient count).
  - Two producers: `buildEnvelopeFromHeaders` (RFC822) and `buildEnvelopeFromFields`
    (Graph — provider strings still treated as untrusted: decoded [no-op when already
    decoded] + capped identically, so the two forms converge).
- **`ParseOut` + the typed result union** gain `displayEnvelope`; populated by
  `hardenedParse` (headers) and the Outlook walker (Graph fields). `buildResultFromParse`
  threads it onto all three `ok` variants. **SafeText subject is now the decoded
  subject in both forms** (was the raw encoded-word header on the RFC822 path) — a
  latent inconsistency fixed in passing; this is also what the live gateway showed.
- **Degradation, not risk routing (§1.2):** a failed/ambiguous decode does NOT
  quarantine. The field shows the capped raw string and is listed in
  `degradedFields`; content processing proceeds. Rationale recorded in the module
  header: nothing structural crosses the boundary unparsed.

## §2 — Orchestrator provider-native bookkeeping (flag-on)

- **IMAP** (`providers/imap.ts`): flag-on, `stream.once('end')` skips `simpleParser`
  entirely and builds the message from provider-native attrs (UID, INTERNALDATE →
  `date`, flags) + the opaque `BODY[]` buffer as `rawRfc822`. `openBox(...,true)` is
  READ-ONLY ⇒ `BODY.PEEK[]` semantics (`\Seen` never set). No `ENVELOPE`/header parse,
  no local header decode. Flag-off path is the original `simpleParser` branch,
  **byte-identical**.
- **Gmail** (`providers/gmail.ts`): flag-on, a single `format=raw` fetch yields
  id/threadId/labelIds/internalDate + opaque raw; `buildRawGmailMessage` assembles the
  message with no header/body walk. Flag-off path = `format=full` + `parseGmailMessage`,
  unchanged.
- **Outlook**: payload per the existing opaque-input union (structured-json default);
  envelope fields (`from`/`toRecipients`/`ccRecipients`/`replyTo`/`receivedDateTime`/
  `internetMessageId`) shipped opaque and decoded in-guest by the walker.
- **`messageRouter`**: plain + carrier rows take from/to/cc/subject/date from the
  guest `displayEnvelope`; bookkeeping date is provider-native (INTERNALDATE/
  internalDate/receivedDateTime), falling back to the guest-decoded `Date` header.

### §2.3 Notification ordering (product-visible delta — for sign-off)

Flag-on, a "new message" notification that fires before depackage completes has **no
subject/sender** (the orchestrator no longer parses headers up-front). Subject/sender
**backfill** when the depackage job returns the `displayEnvelope`. Net delta: a brief
window where a newly-arrived mail shows a generic "new message" before its
subject/sender populate. No correctness impact; flag-off unchanged. **Sign-off
required** on this UX delta before enabling the flag in production.

## Deviations (reported, not absorbed)

1. **Provider FETCH refactors are flag-gated and rig-validated, not off-rig.** The IMAP
   `BODY[]`-only and Gmail `format=raw`-only paths run only flag-on, which has no
   off-rig harness with a live mailbox. They are verified by inspection + the guard
   tripwires + flag-off byte-identity; **V3/V4 on the rig are the acceptance gate.**
2. **`threadingHints` extends the guest result beyond the spec's literal §1.1 field
   list.** Spec §1.1 enumerates the display fields (subject/from/to/cc/reply-to/date);
   Message-ID is a threading key, not a display field. To avoid an IMAP threading/MOVE
   usability regression flag-on (and per §2.2's "from the guest's displayEnvelope/
   **result**"), Message-ID/In-Reply-To/References are derived **in-guest** as a separate
   `threadingHints` field on the result. This keeps header handling out of the
   orchestrator while preserving threading — chosen for "usability AND security maximum
   priority." Flagged here rather than silently absorbed.
3. **Notification-ordering UX delta** (§2.3) requires product sign-off before the flag
   is enabled.
4. **DB-backed flag-on storage parity** (envelope columns end-to-end) is asserted by the
   unit + equivalence corpus and the `messageRouter` wiring; the sqlite-backed
   end-to-end legs remain `skipIf(!Database)` off-rig, same as B2/B2.1.
